# PACademy Admin Backend

Internal-network admin API. Owns DDL/migrations for the shared SQL Server
DB; manages cycles, lookups (24 typed), payments ledger, grades import,
committees, applicant review queues.

**Out of scope for this backend**: applicant journey (drafts, stages,
family, payment intent, exam pick). Those live in the **Applicant
backend** (`../applicant/`) which runs on a separate host and shares
this DB read-only.

> 📘 Architecture decisions, module split, error envelope, and the
> rationale for the two-backend topology live in
> `/Users/mac/.claude/plans/foamy-watching-map.md` (the approved plan).

## 🚨 Seed-data rule (non-negotiable)

**Every table seed must mirror the full frontend mock dataset verbatim — no subsets, no invented rows.** The client reviewed and approved the mock as real production-day data; partial or invented seeds break data integrity and the demo narrative.

Authoritative mock sources (copy row-for-row into seeders):

| Domain | Mock file |
|---|---|
| 24 typed lookups (faculties, specializations, universities, governorates, jobs, …) | `frontend/src/features/lookups/mock/lookups.mock.ts` |
| Applicants + cycles + categories + payments | `frontend/src/shared/mock-data/index.ts` (+ helpers it imports) |
| MOI verification + 4 demo NIDs | `frontend/src/features/applicant-portal/lib/moi-session.mock.ts` |
| Applicant draft / exam slots / payment transactions | `frontend/src/shared/mock-data/applicantPortal.ts` |

**When the remote DB already has a table matching the one you're building** (orphan from a wiped branch, different schema) → **drop it** and recreate with the new schema + full mock seed. Don't try to migrate the old data; the schemas differ and the mock is the source of truth.

**Never POST throwaway test rows against the live DB** during smoke testing — delete them immediately or test against a local SQLite/InMemory instance instead.

## TL;DR — get it running

```bash
# from backend/ (where the .slnx files live)
dotnet build PACademy.Admin.slnx
dotnet run   --project admin/PACademy.Admin.Api --urls http://localhost:5101
```

On first boot the app:
1. Reads `ConnectionStrings:Default` from `admin/PACademy.Admin.Api/appsettings.json`.
2. Calls `Database.Migrate()` to bring the shared SQL Server DB up to schema.
3. Seeds any empty lookup tables with their canonical row set.

Open these URLs after startup:
- `http://localhost:5101/api/lookups/faculties` — sample endpoint (GET)
- `http://localhost:5101/scalar` — interactive API docs (Scalar, replaces Swagger)
- `http://localhost:5101/openapi/v1.json` — raw OpenAPI spec

To skip the auto-migration on startup (e.g. when running multiple admin
instances): `dotnet run --no-seed` or set `SkipMigrationsAndSeed=true` in
the environment.

Each module follows the same four-tier shape:

| Sub-project | Contains |
|---|---|
| `.Domain` | Pure entity classes + enums (lives in `shared/` for canonical types) |
| `.Application` | Use cases (sealed classes), DTO records, FluentValidation validators, `I<Module>DbContext` interface |

| `.Public` (optional) | `I<Module>Api` interface exposed to other modules for cross-module calls (none needed yet) |

## How to add a new endpoint — recipe

Walk-through for adding a new admin endpoint, using `POST /api/lookups/specializations` as a hypothetical example.

### 1. Define / reuse the entity in `shared/PACademy.Shared.Domain/Lookups/`

```csharp
public sealed class Specialization
{
    private Specialization() { }
    public string Code { get; private set; } = default!;
    public string Name { get; private set; } = default!;
    public string FacultyCode { get; private set; } = default!;   // FK
    public bool IsActive { get; private set; }
    public DateTimeOffset CreatedAt { get; private set; }
    public DateTimeOffset UpdatedAt { get; private set; }
    public byte[] RowVersion { get; private set; } = default!;

    public static Specialization Create(string code, string name, string facultyCode)
    {
        // …guards…
        return new Specialization { Code = code, Name = name, FacultyCode = facultyCode,
            IsActive = true, CreatedAt = DateTimeOffset.UtcNow, UpdatedAt = DateTimeOffset.UtcNow,
            RowVersion = [] };
    }
}
```

### 2. Add EF configuration in `shared/PACademy.Shared.Persistence/Lookups/`

```csharp
public sealed class SpecializationConfiguration : IEntityTypeConfiguration<Specialization>
{
    public void Configure(EntityTypeBuilder<Specialization> b)
    {
        b.ToTable("specializations");
        b.HasKey(x => x.Code);
        b.Property(x => x.Code).HasColumnName("code").HasMaxLength(16);
        b.Property(x => x.Name).HasColumnName("name").HasMaxLength(120).IsRequired();
        b.Property(x => x.FacultyCode).HasColumnName("faculty_code").HasMaxLength(16).IsRequired();
        b.Property(x => x.IsActive).HasColumnName("is_active");
        b.Property(x => x.CreatedAt).HasColumnName("created_at");
        b.Property(x => x.UpdatedAt).HasColumnName("updated_at");
        b.Property(x => x.RowVersion).HasColumnName("row_version").IsRowVersion();
    }
}
```

### 3. Expose on the module's `I<Module>DbContext`

```csharp
public interface ILookupsAdminDbContext
{
    DbSet<Faculty> Faculties { get; }
    DbSet<Specialization> Specializations { get; }     // ← new
    Task<int> SaveChangesAsync(CancellationToken ct);
}
```

…and on the concrete `LookupsAdminDbContext`.

### 4. Write the use case (sealed class, primary-ctor DI)

```csharp
public sealed class CreateSpecializationUseCase(ILookupsAdminDbContext db)
{
    public async Task<(SpecializationAdminDto? Ok, string? ErrorCode)> ExecuteAsync(
        CreateSpecializationRequest request,
        CancellationToken ct = default)
    {
        if (await db.Specializations.FindAsync([request.Code], ct) is not null)
            return (null, ErrorCodes.LookupCodeDuplicate);

        // (optional) FK guard
        if (await db.Faculties.FindAsync([request.FacultyCode], ct) is null)
            return (null, ErrorCodes.LookupKeyUnknown);

        var entity = Specialization.Create(request.Code, request.Name, request.FacultyCode);
        db.Specializations.Add(entity);
        await db.SaveChangesAsync(ct);

        return (Map(entity), null);
    }
}
```

### 5. Add a FluentValidation validator

```csharp
public sealed class CreateSpecializationValidator : AbstractValidator<CreateSpecializationRequest>
{
    public CreateSpecializationValidator()
    {
        RuleFor(x => x.Code).NotEmpty().Matches("^SPC-[0-9]{2,4}$");
        RuleFor(x => x.Name).NotEmpty().MaximumLength(120);
        RuleFor(x => x.FacultyCode).NotEmpty().Matches("^FAC-[0-9]{2,4}$");
    }
}
```

### 6. Register the use case + validator in the module DI extension

```csharp
public static IServiceCollection AddLookupsAdminModule(this IServiceCollection s, IConfiguration c)
{
    // …existing wire-up…
    s.AddScoped<CreateSpecializationUseCase>();
    s.AddScoped<IValidator<CreateSpecializationRequest>, CreateSpecializationValidator>();
    return s;
}
```

### 7. Add the controller (in `admin/PACademy.Admin.Api/Controllers/`)

```csharp
[ApiController]
[Route("api/lookups/specializations")]
public sealed class SpecializationsController(
    ListSpecializationsUseCase list,
    CreateSpecializationUseCase create,
    IValidator<CreateSpecializationRequest> createValidator) : ControllerBase
{
    [HttpGet] public async Task<IActionResult> List([FromQuery] string? facultyCode, CancellationToken ct)
        => Ok(await list.ExecuteAsync(new ListSpecializationsFilters(facultyCode), ct));

    [HttpPost] public async Task<IActionResult> Create([FromBody] CreateSpecializationRequest body, CancellationToken ct)
    {
        var v = await createValidator.ValidateAsync(body, ct);
        if (!v.IsValid)
            return BadRequest(new { code = ErrorCodes.ValidationFailed,
                errors = v.Errors.ToDictionary(e => char.ToLowerInvariant(e.PropertyName[0]) + e.PropertyName[1..], e => e.ErrorMessage) });

        var (ok, errorCode) = await create.ExecuteAsync(body, ct);
        if (errorCode == ErrorCodes.LookupCodeDuplicate)
            return Conflict(new { code = ErrorCodes.Conflict, conflictCode = errorCode, message = "كود التخصص مستخدم مسبقاً." });
        if (errorCode == ErrorCodes.LookupKeyUnknown)
            return UnprocessableEntity(new { code = errorCode, message = "كلية غير معروفة." });
        return CreatedAtAction(nameof(List), new { code = ok!.Code }, ok);
    }
}
```

### 8. Generate a migration + restart

```bash
# from backend/
dotnet ef migrations add AddSpecializations \
  --project admin/Modules/LookupsAdmin/PACademy.Modules.LookupsAdmin.Infrastructure \
  --startup-project admin/PACademy.Admin.Api \
  --context LookupsAdminDbContext \
  --output-dir Migrations

dotnet run --project admin/PACademy.Admin.Api --urls http://localhost:5101
# migration auto-applies on startup
```

That's it — 7 files (entity, config, dto, use case, validator, controller line, DI line) + one migration. Every other admin endpoint follows the same shape; if any deviates, justify in the PR description.

## Conventions (non-negotiable)

| Concern | Rule |
|---|---|
| Use case shape | `public sealed class XxxUseCase(IDbContext db, ...deps)` with `Task<...> ExecuteAsync(...request, CancellationToken ct = default)`. Returns `(Ok, ErrorCode)` tuple when the operation can fail predictably; throws only for unexpected conditions. |
| DTOs | `sealed record` only — never classes |
| Mutating methods on entities | Live on the entity (not in use cases). Use case calls `entity.Method(...)`; entity validates internally. |
| Validation | FluentValidation `AbstractValidator<TRequest>`; controller resolves via DI and runs before calling the use case. Errors mapped to `field: message` dictionary. |
| Error envelope | Conflict → `{ code: "CONFLICT", conflictCode: "<TYPED>", payload?: ..., message: "<Arabic>" }`. Use `ErrorCodes.*` from `PACademy.Shared.Contracts`. Never rename a constant after it ships. |
| DB columns | Snake_case via `HasColumnName("...")`. Always include `created_at`, `updated_at`, `row_version`. |
| Concurrency | All mutable rows have `IsRowVersion()` on a `byte[]` property called `RowVersion`. |
| Auth | Every endpoint `[Authorize(Policy = "resource:verb")]`. Resource string matches the URL prefix. (Identity module + policies are not yet wired — placeholder until then.) |
| Audit | Every write emits `IAuditApi.RecordAsync(source: "admin", ...)`. (Audit module not yet wired — placeholder.) |
| CORS | Origin lives in `appsettings.json → Cors:AdminFrontendOrigin`. Never hardcode. |

## Working with EF migrations

The **admin backend owns all migrations**. The applicant backend never runs `dotnet ef database update` — it reads through `IQueryable` only.

```bash
# Add a new migration after changing an entity / config
dotnet ef migrations add <Name> \
  --project admin/Modules/LookupsAdmin/PACademy.Modules.LookupsAdmin.Infrastructure \
  --startup-project admin/PACademy.Admin.Api \
  --context LookupsAdminDbContext \
  --output-dir Migrations

# Apply migrations manually (admin backend also does this on startup)
dotnet ef database update \
  --project admin/Modules/LookupsAdmin/PACademy.Modules.LookupsAdmin.Infrastructure \
  --startup-project admin/PACademy.Admin.Api \
  --context LookupsAdminDbContext

# Rollback one migration
dotnet ef migrations remove \
  --project admin/Modules/LookupsAdmin/PACademy.Modules.LookupsAdmin.Infrastructure \
  --startup-project admin/PACademy.Admin.Api \
  --context LookupsAdminDbContext
```

The design-time factory (`LookupsAdminDbContextFactory`) reads the connection string from `admin/PACademy.Admin.Api/appsettings.json` (or `EFCORE_CONNECTION_STRING` env var) so EF tooling doesn't need to boot the host.

Each module gets its own `__EFMigrationsHistory_<Module>` table — they don't clash.

## Connection string

In `admin/PACademy.Admin.Api/appsettings.json`:

```json
{
  "ConnectionStrings": {
    "Default": "Server=...;Database=PACademy;User Id=...;Password=...;TrustServerCertificate=True;"
  }
}
```

Override per environment by either:
- Adding `appsettings.Production.json` (read automatically),
- Setting `ConnectionStrings__Default` env var (overrides JSON),
- Using `dotnet user-secrets set ConnectionStrings:Default "..."` in dev.

Never commit secrets — production should ride env vars or a vault.

## UAT database copy and switching

The backend can switch between named SQL Server connection strings without code changes:

```bash
export Database__ActiveConnectionName=AdminDbUat
export Database__Schema=PACademy_staging_db
export ConnectionStrings__AdminDbUat='Server=...;Database=PACademy_Admin_UAT;User Id=...;Password=...;TrustServerCertificate=True'
dotnet run --project backend/admin/PACademy.Admin.Api/PACademy.Admin.Api.csproj --urls http://localhost:5101
```

Supported overrides:

- `Database__ActiveConnectionName` or `ADMIN_DB_CONNECTION_NAME` selects the connection-string key.
- `Database__Schema` or `ADMIN_DB_SCHEMA` selects the SQL Server schema, for example `admin_v2` or `PACademy_staging_db`.
- `ConnectionStrings__AdminDb` / `ConnectionStrings__AdminDbUat` hold environment-specific SQL Server strings.
- `ADMIN_DB_CONNECTION_STRING` can override the selected connection string directly for one-off backend runs.

The `/health/db` endpoint returns the active `connectionName`, schema, provider, and whether the backend is using InMemory. It never returns the connection string.

Option A, separate UAT database: create an exact UAT copy of the current admin DB with backup/restore:

```bash
sqlcmd -S <server> -U <user> -P '<password>' \
  -i backend/admin/scripts/copy-admin-db-to-uat.sql \
  -v SourceDb=PACademy_Admin \
     TargetDb=PACademy_Admin_UAT \
     BackupPath=/tmp/PACademy_Admin_UAT_COPY.bak \
     SourceLogicalDataName=PACademy_Admin \
     SourceLogicalLogName=PACademy_Admin_log \
     TargetDataFile=/var/opt/mssql/data/PACademy_Admin_UAT.mdf \
     TargetLogFile=/var/opt/mssql/data/PACademy_Admin_UAT_log.ldf
```

Use the logical file names returned by `RESTORE FILELISTONLY` if your database files are named differently. The script uses `COPY_ONLY` backup/restore so it copies the current schema, EF migration history, constraints, indexes, rowversion columns, and all data into the UAT database.

Option B, separate UAT schema in the same database:

```bash
ASPNETCORE_ENVIRONMENT=Uat \
Database__Schema=PACademy_staging_db \
ConnectionStrings__AdminDbUat='Server=...;Database=PACademy_Admin;User Id=...;Password=...;TrustServerCertificate=True' \
dotnet run --project backend/admin/PACademy.Admin.Api/PACademy.Admin.Api.csproj --urls http://localhost:5101
```

Stop the backend after migrations create the `PACademy_staging_db` schema, then copy current data from `admin_v2`:

```bash
sqlcmd -S <server> -U <user> -P '<password>' -d PACademy_Admin \
  -i backend/admin/scripts/copy-admin-schema-to-uat-schema.sql \
  -v SourceSchema=admin_v2 TargetSchema=PACademy_staging_db
```

Run the UAT backend as a separate service with `ASPNETCORE_ENVIRONMENT=Uat`, `Database__ActiveConnectionName=AdminDbUat`, `Database__Schema=PACademy_staging_db`, and its own CORS origins.
For a staging schema that was already created and copied from prod, set `SkipMigrationsAndSeed=true` so startup does not try to rerun EF migrations against the runtime-switched schema.

Deploy the UAT frontend as a separate Vercel project using:

```bash
vercel --local-config vercel.uat.json
```

Set the UAT Vercel environment variable `VITE_API_BASE_URL=https://admin-staging-api.appenzademo.com` and keep `VITE_USE_MOCKS=false`.
